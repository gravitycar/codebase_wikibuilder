# codebase_wikibuilder

`codewiki` is an LLM-powered CLI that scans a target codebase and builds a searchable Obsidian wiki from it. Each source file gets a markdown summary page. Summaries are grouped into directory overviews, cross-linked, and indexed so you can ask natural language questions and get grounded answers — either from the CLI or from Claude Code via MCP.

The wiki lives in an Obsidian vault you control. `codewiki` only ever adds, updates, or removes files inside that vault; it never modifies your source code.

---

## How it works

1. **Ingest** — scans your codebase, summarizes each file via LLM, and writes `<filename>.<ext>.md` summary pages into the vault. Uses MD5 hashing so only changed files are re-summarized on subsequent runs.
2. **Analysis** — reads all existing summaries and writes `overview.md` files at each directory level, synthesizing a high-level picture of the codebase.
3. **Query** — answers a natural language question by finding relevant summaries and sending them to the LLM, then offers to save the answer as a query page.
4. **Lint** — three-stage maintenance: re-runs stale query pages, deduplicates near-identical ones, and runs a deep health-check that produces `lint-report.md`.

---

## Requirements

- Python 3.10+
- [uv](https://docs.astral.sh/uv/) (package manager)
- An [Anthropic](https://console.anthropic.com/) or [OpenAI](https://platform.openai.com/) API key
- An [Obsidian](https://obsidian.md/) vault (any empty directory works)

---

## Installation

Clone this repo and install into a uv-managed environment:

```bash
git clone <repo-url> ~/codewiki
cd ~/codewiki
uv sync
```

After installation, `codewiki` and `wiki-mcp` are available via `uv run`:

```bash
uv run codewiki --help
```

## Setup

### 1. Create (or choose) your Obsidian vault

Your vault is the directory where all wiki pages will be written. It can be an existing Obsidian vault or a fresh empty directory — `codewiki` will create the necessary structure on first ingest.

```
/path/to/my-vault/        ← vault root
  .wiki-config.json       ← created on first ingest
  .env                    ← you create this (API key)
  index.md                ← auto-generated
  overview.md             ← auto-generated by `analysis`
  src/
    auth.py.md
    database.py.md
  queries/
    how-does-auth-work.md
```

### 2. Add your API key

Create a `.env` file in the vault root:

```bash
# For Anthropic (default)
echo "ANTHROPIC_API_KEY=sk-ant-..." >> /path/to/my-vault/.env

# For OpenAI
echo "OPENAI_API_KEY=sk-..." >> /path/to/my-vault/.env
```

### 3. Run your first ingest

On first run, `codewiki ingest` detects the missing config file and prompts you for the path to your target codebase:

```bash
uv run codewiki ingest --vault /path/to/my-vault
```

```
No configuration file found. Let's set up your wiki.
Config will be saved to: /path/to/my-vault/.wiki-config.json

Enter the absolute path to your target codebase: /path/to/my-project
Configuration saved to /path/to/my-vault/.wiki-config.json

Phase 1: Scanning codebase for changes…
  Found: 42 new, 0 modified, 0 deleted, 0 unchanged

Phase 2a: Summarizing 42 file(s)…
  [████████████████████] 42/42

Phase 2b: No deletions.
Phase 2c: Rebuilding index.md…
Phase 2d: Checking query pages for staleness…

Ingest complete.
 Category                   Count
 Files summarized              42
   New                         42
   Modified                     0
 Files skipped (unchanged)      0
 Files skipped (binary)         3
 Files skipped (too large)      0
 Files deleted                  0
 Files failed                   0
```

The generated `.wiki-config.json` looks like this (edit it to change the model or thresholds):

```json
{
  "codebase_path": ["/path/to/my-project"],
  "llm_provider": "anthropic",
  "llm_model": "claude-sonnet-4-6",
  "file_size_threshold": 100000,
  "inter_request_delay": 1.0
}
```

To ingest multiple directories or specific files, use an array with multiple entries:

```json
{
  "codebase_path": [
    "/path/to/my-project/src",
    "/path/to/my-project/frontend",
    "/path/to/my-project/config.php"
  ]
}
```

Legacy single-string values (`"codebase_path": "/some/dir"`) are still accepted and automatically converted to a one-element list.

| Field | Description | Default |
|---|---|---|
| `codebase_path` | List of absolute paths (files and/or directories) to ingest | (required) |
| `llm_provider` | `"anthropic"` or `"openai"` | `"anthropic"` |
| `llm_model` | Model name string | `"claude-sonnet-4-6"` |
| `file_size_threshold` | Max file size in bytes; larger files are skipped | `100000` |
| `inter_request_delay` | Seconds to wait between LLM requests | `1.0` |

## CLI reference

All commands accept `--vault PATH` (or `-v PATH`) to specify the vault root. If omitted, the current directory is used.

### `codewiki ingest`

Scan the codebase and update wiki summaries.

```bash
# Ingest from the current directory (vault = cwd)
uv run codewiki ingest

# Ingest with an explicit vault path
uv run codewiki ingest --vault /path/to/my-vault
```

Re-running ingest is safe and fast — only files whose content has changed (detected by MD5) are re-summarized.

**Exit codes:** `0` success · `1` fatal error · `2` partial success (some files failed)

---

### `codewiki analysis`

Read all current summary pages and write `overview.md` files at each directory level.

```bash
uv run codewiki analysis --vault /path/to/my-vault
```

Run this after a large ingest to get a high-level synthesis of the codebase. The root `overview.md` is suitable for sharing with an AI assistant as a codebase primer.

**Exit codes:** `0` success · `1` fatal error (run `ingest` first if index.md is missing)

---

### `codewiki query`

Ask a natural language question and get an answer grounded in your wiki.

```bash
uv run codewiki query "How does authentication work?" --vault /path/to/my-vault

uv run codewiki query "What database models exist and how are they related?"
```

The answer always ends with a `## Sources` section listing which summary pages were consulted. After printing the answer, the CLI prompts whether to save it (default: No):

```
Save this answer to the wiki? [N]:
```

If you save, the answer is written to `queries/<slug>.md` and added to `index.md`.

**Exit codes:** `0` success · `1` fatal error · `3` no relevant summaries found

---

### `codewiki lint`

Run three-stage vault maintenance: resolve stale query pages, deduplicate near-duplicates, and write a health-check report.

```bash
uv run codewiki lint --vault /path/to/my-vault
```

Run `lint` periodically (e.g., after several ingests) or whenever `ingest` warns that query pages are stale. The output includes a `lint-report.md` in the vault root covering orphans, missing cross-references, and concept gaps.

**Exit codes:** `0` success · `1` fatal error

---

### `codewiki help`

Print detailed usage information for any command or topic.

```bash
uv run codewiki help
uv run codewiki help ingest
uv run codewiki help query
uv run codewiki help mcp
```

## Typical workflow

```bash
# 1. First-time setup (prompts for codebase path)
uv run codewiki ingest --vault ~/my-vault

# 2. Generate directory overviews
uv run codewiki analysis --vault ~/my-vault

# 3. Ask questions
uv run codewiki query "How is the payment flow implemented?" --vault ~/my-vault

# 4. Re-ingest after code changes (only changed files are re-processed)
uv run codewiki ingest --vault ~/my-vault

# 5. Periodic maintenance
uv run codewiki lint --vault ~/my-vault
```

## MCP server (Claude Code integration)

`wiki-mcp` is a stdio MCP server that exposes a `wiki_query` tool to Claude Code. With it, Claude can query your wiki directly during a coding session without leaving the editor.

**Claude Code starts the MCP server automatically** when it reads `.mcp.json` — you do not need to start it manually.

### Setup

Add a `.mcp.json` file to the root of your **target codebase** (not the vault):

```json
{
  "mcpServers": {
    "wiki": {
      "command": "uv",
      "args": ["run", "--project", "/path/to/my-vault", "wiki-mcp"]
    }
  }
}
```

Replace `/path/to/my-vault` with the absolute path to your Obsidian vault.

Then restart Claude Code. It will discover the `wiki_query` tool automatically. You can also run `uv run codewiki help mcp` from inside the vault to get a pre-filled `.mcp.json` snippet for that vault.

### Tool schema

```
Tool:   wiki_query
Input:  { "question": "string" }
Output: {
  "answer":        "string",
  "sources":       ["string"],
  "saved_path":    "string",
  "stale_warning": ["string"] | null
}
```

The MCP server always saves answers automatically (no interactive prompt). Maintenance commands (`ingest`, `analysis`, `lint`) are CLI-only and are not exposed via MCP.

### Example `.mcp.json` for multiple codebases

If you maintain separate wikis for separate projects, you can name each server differently:

```json
{
  "mcpServers": {
    "backend-wiki": {
      "command": "uv",
      "args": ["run", "--project", "/vaults/backend", "wiki-mcp"]
    },
    "frontend-wiki": {
      "command": "uv",
      "args": ["run", "--project", "/vaults/frontend", "wiki-mcp"]
    }
  }
}
```

## Vault structure

After a full run the vault looks like this:

```
my-vault/
  .wiki-config.json       config (codebase path, LLM settings)
  .env                    API key (ANTHROPIC_API_KEY or OPENAI_API_KEY)
  index.md                master index of all pages
  overview.md             root-level synthesis (written by `analysis`)
  log.md                  append-only run history
  logs/
    2026-01-15_10-30-00.log
  src/
    overview.md           directory-level synthesis
    auth.py.md
    database.py.md
    models/
      overview.md
      user.py.md
  queries/
    how-does-auth-work.md
    what-database-models-exist.md
  lint-report.md          written by `codewiki lint`
```

Summary files are named `<filename>.<ext>.md` (e.g., `auth.py.md`) and mirror the codebase directory structure. Binary files, files over `file_size_threshold` bytes, and files in excluded directories (`.git`, `.venv`, `node_modules`, `__pycache__`, etc.) are skipped automatically.
